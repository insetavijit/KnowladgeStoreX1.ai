| :--- | :--- | :--- | :--- |
| **[[1.4.2.1 Custom Tags]]** | Delimiters | `<thought>...</thought>`, segmentation, parsable blocks | Custom tags are often superior to JSON for separating different *types* of content (reasoning vs answer) in one response. |
| **[[1.4.2.2 Attributes vs Content]]** | Metadata injection | `<step id="1">`, metadata storage, concise labeling | Attributes allow you to attach meta-information (IDs, timestamps) to content without cluttering the text. |

| **[[1.4.2.4 Markdown Structures]]** | Document formatting | Headers, tables, bolding, blockquotes | Markdown is the native language of LLMs, making it the most reliable format for human-readable structured text. |
| **[[1.4.2.5 YAML Blocks]]** | Config data | Indentation-based, frontmatter, readability | YAML is often more token-efficient and readable than JSON for configuration files or frontmatter. |
| **[[1.4.2.6 Tag Validation]]** | Closing tags | Well-formed XML, regex checkers, loose parsing | Ensuring every opening tag has a corresponding closing tag is critical for regex-based extraction. |
| **[[1.4.2.7 Extracting Tagged Content]]** | Parsing | Regex capture groups, BeautifulSoup, parsing logic | Extracting content between `<tags>` is robust against minor formatting errors that would break JSON. |
| **[[1.4.2.8 Nesting & Indentation]]** | Hierarchy | Tree structures, parent-child relationships, scope | XML handles deep nesting intuitively, making it good for representing document hierarchies. |
| **[[1.4.2.9 CDATA Handling]]** | Raw text | Encoding special chars, literal blocks, code inclusion | Understanding how to safely include code or raw text inside XML tags prevents parsing breaks. |
| **[[1.4.2.10 RSS/Atom Feeds]]** | Syndication | Standard schemas, feed generation, podcast meta-data | Generating standard XML feeds allows LLM content to be syndicated across existing web ecosystems. |
